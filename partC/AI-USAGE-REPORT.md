# AI ашиглалтын тайлан

Энэ тайлан нь Mini Library Management System төслийг AI-ийн тусламжтайгаар төлөвлөж, хэрэгжүүлж, шалгаж, баримтжуулах явцад AI ямар үүрэг гүйцэтгэсэн, би өөрөө ямар шийдвэр гаргаж ямар хяналт хийсэн талаар тайлбарлана. Тайлангийн зорилго нь AI ашигласан гэдгийг нуух биш, харин AI-ийн туслалцаа бодитоор хаана хэрэг болсон, хаана эрсдэл үүсгэсэн, түүнийг би яаж шалгаж зассан, мөн өөрийн суралцах чадвараа алдахгүй байхын тулд юу хийсэн гэдгийг үнэн зөв харуулах юм.

## 1. Юуг AI хийсэн, юуг би өөрөө хийсэн бэ?

AI хамгийн түрүүнд төслийн scope тодорхойлоход тусалсан. Даалгаврын сэдвүүдээс Mini Library-г сонгох боломж байсан ч би энгийн CRUD төслөөс илүү бодит full-stack бүтэцтэй ажил хийхийг хүссэн. AI номын inventory, гишүүний бүртгэл, зээлэлт үүсгэх, буцаах, хугацаа хэтэрсэн зээлэлтийг харах, dashboard summary, search/filter гэсэн боломжуудыг санал болгосон. Энэ санал нь төслийн үндсэн зураглалыг гаргахад хэрэг болсон. Гэхдээ эцсийн scope-ийг би өөрөө хязгаарласан. Жишээлбэл AI-ийн санал болгож болох торгууль, barcode scanner, олон салбарын дэмжлэг, online payment зэрэг илүү том feature-үүдийг даалгаврын хугацаа, тайлбарлах боломж, implementation эрсдэлийг харгалзан оруулаагүй.

Part A дээр AI planning document-уудын эхний draft-ийг гаргахад тусалсан. `PROJECT.md`, `ARCHITECTURE.md`, `STACK-COMPARISON.md`, `adr/0001-stack-decision.md`, `ai-sessions/plan.md` зэрэг файлын эхний бүтэц, гарчиг, гол санааг AI-аар хурдан бэлдүүлсэн. Миний хийсэн ажил бол эдгээрийг даалгаврын шаардлагатай тулгаж шалгах, Part A-д production code оруулахгүй байх, архитектурын diagram нь бодит implementation-тэй нийцэх эсэхийг нягтлах байсан. Жишээлбэл stack comparison дээр AI React + Express + PostgreSQL, React + NestJS + PostgreSQL, React + Django REST Framework + PostgreSQL гэсэн гурван хувилбарын давуу болон сул талыг тайлбарласан. Би NestJS болон Django REST Framework-ийг сонгоогүй, учир нь энэ даалгаварт Express нь хангалттай ойлгомжтой, жижиг, нэг JavaScript ecosystem-д багтах шийдэл байсан.

Part B дээр AI code scaffold болон давтагдсан boilerplate үүсгэхэд их тусалсан. React frontend-ийн dashboard, books, members, loans page-ийн эхний layout, Express API route-ууд, Prisma schema, OpenAPI file, unit test-ийн эхний хувилбар зэрэг нь AI-ийн тусламжтай хурдан гарсан. Гэхдээ би зөвхөн үүсгэсэн кодыг шууд хүлээж аваагүй. Би route бүрийн behavior-ийг шалгаж, request body validation хэрэгтэй эсэхийг харж, зээлэх боломжийн дүрэм, due date тооцоолол, буцаах үед inventory нэмэгдэх logic, dashboard count зөв эсэхийг тестээр баталгаажуулсан. Жишээлбэл зээл үүсгэхэд `availableCopies` нэгээр буурах, буцаахад нэгээр нэмэгдэх ёстой гэсэн rule-ийг repository transaction test-ээр тусад нь шалгадаг болгосон.

AI shared business rule-үүдийг тусад нь гаргах санааг мөн дэмжсэн. Гэхдээ `partB/src/shared/libraryRules.js` дотор ямар rule орох, ямар нь repository түвшинд үлдэхийг би өөрөө ялгасан. `canLoanBook`, `createLoanRecord`, `extendLoanRecord`, `returnLoanRecord`, `computeLoanStatus`, `summarizeDashboard` зэрэг pure function-ууд нь database ажиллуулахгүйгээр unit test хийх боломжтой. Энэ нь практик шалгалтын үед кодыг тайлбарлахад их чухал. Харин Prisma transaction буюу өгөгдлийн сангийн update дараалал нь repository layer-д үлдсэн.

Part C дээр AI reflection болон AI usage report-ийн draft бичихэд тусалсан. Миний үүрэг нь тайланг бодит хийсэн ажилтай нийцүүлэх, хэт ерөнхий магтаал болгохгүй байх, hallucination болон security risk-ийн бодит жишээ нэмэх байсан. Мөн би бүх markdown файлыг Монгол хэл рүү хөрвүүлэхийг хүсэж, command, path, API endpoint, env variable зэрэг техникийн literal хэсгийг хэвээр үлдээх шаардлагыг баримталсан.

## 2. Hallucination-ийн 2-оос дээш жишээ, яаж олж зассан бэ?

Эхний hallucination эрсдэл нь repository болон Git орчинтой холбоотой байсан. AI ихэвчлэн хэрэглэгчийн хэлсэн repository шууд одоогийн working directory гэж таамаглах магадлалтай. Энэ төсөл дээр local workspace эхэндээ өөр parent repository-д хамаарах боломжтой байсан. Хэрэв Git root, remote, status шалгахгүйгээр commit хийсэн бол өөр repository-д өөрчлөлт орох эрсдэлтэй. Би үүнийг `git status`, `git remote`, working directory шалгах command-уудаар баталгаажуулж, зөв `lab13` folder дотор ажиллаж байгаа эсэхийг нягталсан. Ингэснээр AI-ийн implicit assumption-д найдахгүй, бодит filesystem болон Git state-д тулгуурласан.

Хоёр дахь hallucination эрсдэл нь dependency болон tooling-ийн хувилбартай холбоотой. AI modern React, Vite, Tailwind, Express, Prisma, PostgreSQL ашиглахыг санал болгосон нь зөв чиглэл байсан ч тухайн package-ийн яг ямар хувилбар installed байгаа, command яг энэ machine дээр ажиллах эсэхийг AI өөрөө мэдэхгүй. Жишээлбэл `npm test` болон `npm run build` анх sandbox дотор `spawn EPERM` алдаагаар унасан. AI үүнийг test эвдэрсэн гэж буруу тайлбарлах магадлалтай байсан. Би output-ыг уншаад алдаа нь code logic биш, restricted shell child process spawn хийхийг хориглосонтой холбоотой гэж ялгасан. Дараа нь зөвшөөрөлтэйгээр command-ыг дахин ажиллуулж test болон build pass болж байгааг баталгаажуулсан.

Гурав дахь hallucination жишээ нь assignment file naming болон structure дээр гарч болох байсан. Зарим төлөвлөлтийн нэршил дээр `0001-stack.md` эсвэл `0001-stack-decision.md` гэх хоёр хувилбар гарч болох нөхцөл байсан. AI ийм үед нэгийг нь дур мэдэн сонгож магадгүй. Би project tree болон илүү тодорхой file name-ийг харгалзан `partA/adr/0001-stack-decision.md` гэж хадгалсан. Энэ нь архитектурын шийдвэрийн агуулгыг илүү тодорхой илэрхийлж байсан.

Дөрөв дэх жишээ нь API behavior дээр гарсан. Буцаах товч дарахад frontend body явуулахгүй PATCH request хийж байсан, харин backend дээр `req.body.returnedAt` гэж шууд уншиж байсан тул `req.body` undefined үед 500 error гарсан. AI эхэндээ энэ төрлийн edge case-ийг бүрэн барьж чадаагүй. Би browser console болон backend route flow-ийг харж, request body байхгүй үед optional chaining хэрэгтэйг олсон. Үүнийг `req.body?.returnedAt ?? new Date()` болгож зассан. Дараа нь test болон build ажиллуулж regression байхгүйг шалгасан.

## 3. Security болон license-ийн анхаарал, AI-аас гарсан кодод security risk байсан уу?

Security-ийн хамгийн чухал анхаарал нь secret leakage байсан. Backend-д `DATABASE_URL`, `JWT_SECRET`, `ADMIN_EMAIL`, `ADMIN_PASSWORD` зэрэг sensitive env variable ашиглагддаг. AI project setup хийхдээ жишээ credential README-д бичих шаардлагатай байж болно, гэхдээ бодит `.env` файлыг commit хийх нь security risk. Би `.env` ignore хийгдсэн эсэхийг шалгаж, README-д `.env.example`-оос local `.env` үүсгэх, production secret-ийг commit хийхгүй байх заавар оруулсан. Docker Compose дээр development default credential байж болох ч production орчинд хүчтэй secret ашиглах ёстой гэж runbook-д тэмдэглэсэн.

SQL injection мөн анхаарах зүйл байсан. Анхны архитектурын түвшинд AI raw SQL эсвэл repository abstraction санал болгож болох байсан. Энэ төслийн implementation Prisma ORM ашиглаж байгаа тул user input-оор SQL string залгахгүй. Хэрэв ирээдүйд raw query хэрэглэх бол parameterized query ашиглах шаардлагатай. Энэ шаардлагыг `CLAUDE.md`, security review log, README troubleshooting хэсэгт мөн тусгасан. Search/filter одоогоор application logic болон Prisma query flow-той нийцэж байгаа бөгөөд user input-ыг SQL statement болгож залгахгүй.

Authentication дээр JWT-г cookie-д хадгалах үед cookie option зөв байх ёстой. AI-аас гарсан эхний санааг би шалгаж, httpOnly cookie ашиглах нь browser JavaScript token унших эрсдэлийг бууруулна гэж үзсэн. `authRules.js` дотор cookie option-ийг pure function болгон салгаж, `httpOnly`, `sameSite`, `secure`, `maxAge`, `path` утгуудыг unit test-ээр шалгасан. Production үед `secure` true болох ёстойг test-д оруулсан.

License-ийн хувьд AI-generated code нь public snippet-тэй төстэй байх эрсдэлтэй. Энэ төслийн код нь нийтлэг Express route, React component, Prisma repository pattern дээр суурилсан жижиг custom implementation учраас том license-тэй snippet хуулсан гэж үзэх үндэслэл бага. Гэхдээ production-д ашиглах бол npm package бүрийн license шалгах шаардлагатай. Би unknown том code block хуулбарлахаас зайлсхийж, implementation-ийг жижиг, тайлбарлахад ойлгомжтой function-уудаар хийсэн.

## 4. Юуг AI-аар хурдан хийсэн бэ? Production benefit юу байсан бэ?

AI хамгийн ихээр planning болон documentation-ийг хурдасгасан. Stack comparison, ADR, architecture overview, AI session summary, README runbook зэрэг бичиг баримт гараар бичвэл их цаг авна. AI draft гаргаснаар би хоосон хуудаснаас эхлэхгүй, харин агуулгыг шалгаж, засаж, төслийн бодит байдалтай нийцүүлэхэд илүү их цаг зарцуулсан. Production benefit талаас энэ нь onboarding document болон runbook хурдан бий болгох давуу талтай. Шинэ хөгжүүлэгч эсвэл шалгагч төслийг хэрхэн setup хийх, login хийх, Prisma Studio нээх, DB reset хийх, backup/restore хийхийг README-ээс шууд харж чадна.

AI implementation boilerplate-ийг мөн хурдасгасан. Express app setup, route structure, frontend page component, Tailwind class ашигласан form/table layout, OpenAPI path зэрэг давтагдсан ажлууд AI-аар хурдан draft болсон. Production benefit нь feature delivery эхний шатанд хурд нэмэх явдал. Жишээлбэл ном, гишүүн, зээлэлт гэсэн core workflow хурдан бүрдсэн тул дараагийн үе шатанд validation, transaction, test, error handling дээр төвлөрөх боломж гарсан.

Testing дээр AI edge case жагсаахад их хэрэг болсон. Inactive member-д зээлүүлэхгүй байх, available copy дууссан номыг зээлүүлэхгүй байх, due date өнгөрсөн бол overdue болох, returned loan dashboard overdue count-д орохгүй байх, invalid loan period 1-365 хоногийн хооронд байх зэрэг case-үүдийг AI-тэй хамт илүү хурдан олсон. Production benefit нь regression багасгах. Одоо `npm test` 26 unit test ажиллуулж auth rules, validation, loan transaction, dashboard summary-г шалгана.

AI smoke test script хийхэд мөн тусалсан. `npm run test:api-smoke` нь live backend дээр health, auth login/me/logout, book/member create, loan create, extend, return, list, dashboard summary шалгадаг. Production-like benefit нь deployment эсвэл demo-ийн өмнө API-ийн гол урсгал амьд байгаа эсэхийг хурдан шалгах боломж юм.

## 5. Юуг AI-аар удаан хийсэн бэ? Бэрхшээл ба антипаттерн

AI-аар удаан болсон нэг хэсэг нь generated output-ыг бодит environment-тэй тулгах ажил байсан. AI command санал болгохдоо тухайн shell sandbox, Windows path, Docker service state, database connection string зэрэг бодит нөхцөлийг мэдэхгүй. Тиймээс command бүрийг ажиллуулж output унших, унавал шалтгааныг ялгах шаардлагатай болсон. `spawn EPERM` алдаа бол үүний тод жишээ. AI энэ алдааг code bug гэж таамаглаж болох байсан, харин бодит шалтгаан нь restricted shell байсан.

Өөр нэг удаашруулсан зүйл нь AI заримдаа scope хэт томруулах хандлагатай. Library system гэдэг topic-д role-based access, fine calculation, barcode scanning, notification, reporting, multi-branch support гэх мэт олон feature нэмж болно. Гэхдээ даалгаврын зорилго нь AI-assisted planning, implementation, testing, reflection-ийг харуулах болохоос enterprise library product бүтээх биш. Би scope-ийг ном, гишүүн, зээлэлт, буцаалт, хугацаа хэтэрсэн төлөв, dashboard summary дээр барьсан. Энэ нь anti-pattern болох “AI санал болгосон бүхнийг хийх” хандлагаас зайлсхийсэн.

Documentation consistency шалгах нь мөн цаг авсан. AI нэг file-д Express route гэж бичээд өөр file-д өөр path дурддаг, эсвэл README command нь package.json-той нийцэхгүй байх эрсдэлтэй. Тиймээс OpenAPI, README, package script, actual route, frontend API call-уудыг давхар шалгах шаардлагатай болсон. Жишээлбэл loan extend нэмэх үед backend endpoint, frontend button, OpenAPI path, test, README smoke flow бүгд нийцэх ёстой байсан.

Мөн Монгол хэл рүү хөрвүүлэх үед AI эсвэл tool encoding output заримдаа mojibake мэт харагдах эрсдэлтэй байсан. File-уудыг UTF-8 агуулгаар хадгалж байгаа эсэх, command/path/code block орчуулагдаж эвдрэхгүй байгаа эсэхийг хайлт хийж шалгасан. Энэ нь AI-аар documentation их хэмжээгээр өөрчлөхөд заавал review хэрэгтэйг харуулсан.

## 6. Skill atrophy эрсдэлийг яаж зохицуулсан бэ? “AI байхгүй” цаг гаргасан уу?

Skill atrophy буюу AI-д хэт найдаж өөрөө ойлгох чадвар сулрах эрсдэлийг би хэд хэдэн аргаар зохицуулсан. Нэгдүгээрт, core business logic-ийг жижиг pure function болгон салгасан. `canLoanBook`, `createLoanRecord`, `extendLoanRecord`, `returnLoanRecord`, `computeLoanStatus`, `summarizeDashboard` зэрэг функцүүдийг би өөрөө уншиж, мөр бүрийн behavior-ийг тайлбарлаж чадна. Энэ нь AI-аар үүсгэсэн том блок кодыг ойлгохгүй ашиглах эрсдэлийг бууруулсан.

Хоёрдугаарт, би test output-ыг өөрөө уншиж алдааны шалтгааныг ялгасан. Test унахад AI-аас шууд “засаад өг” гэж найдах биш, алдаа нь validation logic уу, shell permission уу, backend route body undefined юу гэдгийг ялгаж харсан. Return endpoint-ийн 500 алдааг засахдаа frontend request body явуулахгүй байгааг ойлгож, backend optional access хийх ёстой гэж дүгнэсэн. Энэ нь debugging-ийн “AI байхгүй” дадлага болсон.

Гуравдугаарт, би зарим шийдвэрийг AI-ийн санал биш өөрийн тайлбарлах чадвараар хэмжсэн. NestJS илүү professional харагдаж болох байсан ч decorator, dependency injection, module structure нэмэгдэх тул би Express сонгосон. Учир нь даалгаврын үед би route, repository, shared rule-ийн урсгалыг илүү сайн тайлбарлаж чадна. Энэ нь архитектурын шийдвэрийг AI-д бүрэн даатгаагүй гэсэн үг.

Дөрөвдүгээрт, би “AI байхгүй” цагийг бага хэмжээгээр гаргасан. Жишээлбэл test failure output унших, Git status шалгах, README runbook-ийн command-уудыг бодит Docker/Prisma setup-тэй нийцүүлэх, API smoke script ямар endpoint дарааллаар шалгахыг тодорхойлох ажлыг өөрөө бодож хийсэн. Цаашид илүү сайн болгохын тулд return-loan endpoint эсвэл шинэ validation rule-ийг AI-гүйгээр scratch-аас дахин бичиж, дараа нь AI-аар review хийлгэх дадлага хийх хэрэгтэй гэж үзэж байна.

Эцэст нь, AI-г суралцах үйл явцыг орлох хэрэгсэл биш, хурд нэмдэг туслах гэж ашигласан. AI draft гаргаж өгсөн ч би command ажиллуулж, test pass эсэхийг шалгаж, security risk хайж, file structure болон documentation consistency-г нягталсан. Энэ арга нь AI ашиглахын зэрэгцээ өөрийн engineering judgment-ээ хадгалах хамгийн бодит арга байсан.

## Дүгнэлт

AI энэ төсөлд planning, drafting, boilerplate implementation, test idea, documentation translation зэрэг олон ажлыг хурдан болгосон. Гэхдээ AI-ийн output бүрийг бодит environment, assignment requirement, security expectation, test result-тэй тулгаж шалгах шаардлагатай байсан. Миний гол үүрэг бол scope сонгох, архитектурын шийдвэрийг ойлгож батлах, generated code-ыг review хийх, алдааг tool output дээр үндэслэн засах, AI ашиглалтыг үнэн зөв тайлагнах явдал байв. Энэ төсөл production library system биш боловч AI-assisted software construction-ийн төлөвлөлт, implementation, verification, reflection гэсэн бүх үе шатыг багтаасан бүрэн даалгаврын жишээ болсон.
